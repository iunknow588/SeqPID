# 中间文档生成实现对齐分析（2026-07-13）

## 1. 分析范围

本次从“生成中间文档/诊断文件”的角度，对照以下设计文档检查两套实现：

- `PID算法与实现.md`
- `金融领域的物理学概念模型.md`
- `比赛详细设计说明书.md`

检查对象：

- Rust 版：`src-rust`

重点不评价最终提交文件本身的预测效果，而评价代码是否能把 PID 分解、物理概念映射、泄漏检查、稳定性判断、市场状态快照等中间过程以可复核文档形式输出。

## 2. 设计目标摘要

设计文档对中间文档的核心要求如下：

1. 正式提交文件与诊断文件分离。
   - `pattern_reco.csv`、`predict_result.csv`、`submit.zip` 仅保留官方字段。
   - PID、市场、规则、验证信息进入独立诊断文件。

2. PID 诊断需要显式表达状态空间契约。
   - `psi_t = psi_t-1 + eta_t`
   - `y_t = x_t · psi_t + epsilon_t`
   - `y_hat_t+1|t = x_t+1|t · psi_t|t-1`

3. 观测量与预测量不得混写。
   - 应区分 `y_observed`、`y_hat_next`、`v_q_observed`、`v_hat_q_next`。

4. 诊断文件需要覆盖工程可审计字段。
   - `data_leakage_check`
   - `m_eff_rank_eligible`
   - `param_stability_flag`
   - `m_slow_method`
   - `thin_trade_window`
   - `cross_symbol_comparable`
   - `domain_mapping_valid_flag`

5. 实时路径不得使用离线平滑器。
   - 若配置中出现 offline 方法，应在诊断中显式标记泄漏风险。

6. 大单/主动方向/规则资金流必须保持有符号语义。
   - 买入为正，卖出为负。
   - `CH_rule_t`、`Q_rule_t`、`R_seed_t`、`signed_large_active_amount`、`signed_mix_qr_amount` 应作为 PID 资金锚点的有符号来源。

## 3. 修改前实现准确性判断

修改前，两套实现已经具备一部分中间文档能力：

- 已输出 `pid_window_params.csv`、`pid_window_contrib.csv`、`pid_tail_diagnostics.csv`。
- 已输出市场层面的 `market_pid_snapshot.csv`、`market_regime_report.md`。
- 批处理路径已有诊断/验证报告，如批次分布、市场 PID 验证、回放验证等。
- 正式提交文件与诊断文件基本分离。

但与设计文档相比仍存在一个明显缺口：缺少设计标准化的 `pid_window_diag.csv` 与 `pid_daily_diag.csv`。已有 `pid_window_params.csv` 和 `pid_window_contrib.csv` 更偏参数/贡献明细，不能完整承载泄漏检查、稳定性、方法边界、样本稀疏、跨股票可比性、提交就绪状态等审计字段。

因此，修改前结论为：两版代码“部分实现”设计目标，但中间诊断文档不完整，不能完全支撑设计文档要求的可复核链路。
