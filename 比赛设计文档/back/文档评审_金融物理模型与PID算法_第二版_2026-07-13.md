## 二、遗留问题：严重缺陷部分修复

### 1. 【严重】预测口径 `x_t+1|t` 的构造规则仍不完整

**位置：** `PID算法与实现.md` §4.0

**已修复部分：**
- 明确区分了历史解释口径（`psi_t` 吸收 `y_t` 后更新）与实时预测口径（`psi_t|t-1` 先验）
- 增加 `data_leakage_check` 检查项（§10.1），要求确认预测时只使用截至 `t` 或 `t-1` 的输入

**仍存问题：**

```
v_q,t+1|t = x_t+1|t · psi_t + epsilon_t+1
```

文档仅说"明确 `x_t+1|t` 的构造规则"，但未给出具体规则。关键缺失：

1. **`U_ch,t` 在 `t` 窗口结束前是否已知？**
   - 若规则流基于窗口内逐笔数据（如5分钟窗口的逐笔成交），则 `t` 窗口结束前 `U_ch,t` 不可知
   - 此时 `x_t+1|t` 中的 `U_ch,t` 必须替换为 `U_ch,t|t-1`（预测值）或直接用 `U_ch,t-1`
   - 文档未明确 `U_*` 的预测策略：是滚动均值？上一窗口值？还是规则层有独立的预测模型？

2. **`psi_t` 的来源未绑定到具体算法**
   - 文档说"使用更新前的先验或滤波预测量"，但未明确是卡尔曼滤波的一步预测还是滚动OLS的最后一个窗口估计
   - 不同算法的 `psi_t|t-1` 性质不同：卡尔曼滤波有明确的先验分布，滚动OLS没有

3. **`epsilon_t+1` 的处理**
   - 预测时 `epsilon_t+1` 未知，文档未说明是否设为零或保留噪声项
   - 若设为零，预测值是条件均值；若保留噪声方差，需输出预测区间

**后果：**
- 工程实现时，不同开发者可能选择不同的 `x_t+1|t` 构造策略，导致结果不可复现
- 比赛提交时，若 `U_*` 的预测策略与训练时不一致，可能引入系统性偏差

**修复建议：**
1. 明确 `U_*` 的可用性时序：若窗口内逐笔数据在窗口结束前不可用，则预测时只能用 `U_ch,t-1`
2. 给出 `x_t+1|t` 的默认构造规则：
   ```
   x_t+1|t = [v_q,t, U_ch,t-1, U_q,t-1, U_retail,t-1, D_driver_t]^T  # 5维模式
   ```
   或若规则流有独立预测：
   ```
   x_t+1|t = [v_q,t, U_ch,t|t-1, U_q,t|t-1, U_retail,t|t-1, D_driver_t]^T
   ```
3. 明确 `psi_t` 的算法绑定：若用卡尔曼滤波，给出状态转移方程和预测步公式；若用滚动OLS，给出窗口长度和权重策略
4. 明确 `epsilon_t+1` 的处理：预测时设为零，输出点预测；或保留方差，输出预测区间

### 2. 【中等】`theta` 的联合稳定性条件仍不完整

**位置：** `PID算法与实现.md` §6.0 + §11

**已修复部分：**
- 增加 `theta_sign_interpretation` 字段，明确 `theta > 0` 偏助推、`theta < 0` 偏抑制
- 工程校验清单（§11）增加："`abs(phi + theta) < 1` 只能作为快速筛查，正式判断应结合特征根是否位于单位圆内"

**仍存问题：**

1. **`abs(phi + theta) < 1` 不是充分条件**
   - 状态方程 `v_t = phi * v_t-1 + theta * (v_t-1 - v_t-2) + ...` 是二阶差分方程
   - 其特征方程为 `r^2 - (phi + theta) * r + theta = 0`
   - 稳定性要求特征根在单位圆内：`|r_1| < 1` 且 `|r_2| < 1`
   - `abs(phi + theta) < 1` 只是必要条件之一，不是充分条件

2. **未给出特征根的具体计算公式**
   - 工程实现需要明确的稳定性判断逻辑，而非模糊的"结合特征根检验"

3. **`theta` 的边界 `[-1, 1]` 与 `phi` 的边界 `[0, 1]` 联合时可能不稳定**
   - 例如 `phi = 0.9`, `theta = 0.9`：特征方程 `r^2 - 1.8r + 0.9 = 0`，根为 `r = 0.9 ± 0.3i`，模 `|r| = sqrt(0.9^2 + 0.3^2) = 0.95 < 1`，稳定
   - 但 `phi = 0.9`, `theta = -0.9`：特征方程 `r^2 - 0r + (-0.9) = 0`，根为 `r = ±sqrt(0.9) ≈ ±0.95`，稳定
   - 边界情况 `phi = 1`, `theta = -1`：特征方程 `r^2 - 0r + (-1) = 0`，根为 `r = ±1`，临界不稳定
   - 文档未分析这些边界组合

**修复建议：**
1. 给出明确的稳定性条件：
   ```
   特征方程：r^2 - (phi + theta) * r + theta = 0
   稳定性要求：|r_1| < 1 且 |r_2| < 1
   等价于：
   (1) |theta| < 1
   (2) |phi + theta| < 1 + theta
   (3) |phi + theta| < 1 - theta  （当 theta < 0 时）
   ```
   或更简洁的 Jury 稳定性判据
2. 在参数边界表中增加联合稳定性列，明确哪些 `(phi, theta)` 组合是允许的
3. 工程实现中增加实时稳定性检查，若参数越界则触发 `param_stability_flag` 并回退到基线

## 三、新增缺陷

### 3. 【中等】`m_slow` 的提取方法未明确

**位置：** `PID算法与实现.md` §6.1 + §10

**问题描述：**
文档说 `m_slow` 可通过"卡尔曼滤波、RTS 平滑或滚动稳健统计提取"，并推荐固定历史窗口长度（20日或60日）。

**逻辑缺陷：**
1. 卡尔曼滤波提取趋势项（`m_slow`）与 RTS 平滑提取趋势项是不同的方法，前者是实时可用的，后者使用了未来信息
2. 文档未明确"卡尔曼滤波趋势项"具体指什么：是状态方程中的 `psi_t` 的某个分量？还是独立的对 `m_eff` 序列做滤波？
3. 滚动稳健统计（如滚动中位数）的窗口长度与固定历史窗口长度（20日/60日）的关系未澄清：是同一概念还是不同方法？
4. 若用 EWMA 提取 `m_slow`，衰减因子 `alpha` 未给出建议值

**后果：**
- 不同实现可能选择不同的 `m_slow` 提取方法，导致跨团队结果不可比
- 比赛提交时，若 `m_slow` 的提取方法与训练时不一致，可能影响评分稳定性

**修复建议：**
1. 明确 `m_slow` 的推荐提取方法优先级：
   - 首选：对 `m_eff` 序列做 EWMA，衰减因子 `alpha = 2 / (N + 1)`，N 为窗口长度
   - 次选：滚动中位数，窗口长度固定为20日或60日
   - 备选：卡尔曼滤波趋势项（需明确状态方程）
2. 明确 RTS 平滑仅用于离线复盘，不得用于实时预测
3. 增加 `m_slow_method` 字段，标记所用方法

### 4. 【中等】`beta_norm` 的归一化口径存在量纲隐患

**位置：** `PID算法与实现.md` §6.1 + `金融领域的物理学概念模型.md` §7.2

**问题描述：**
`beta_norm` 定义：
```
beta_norm_* = (capital_* / P) / (U_* / market_value)
```

文档说 `U_* / market_value` 必须是无量纲口径。

**逻辑缺陷：**
1. `capital_* = beta_* * U_*`，若 `U_*` 是规则分数（无量纲），则 `capital_*` 的量纲与 `beta_*` 相同，但 `capital_* / P` 要求 `capital_*` 是金额量纲，矛盾
2. 若 `U_*` 是金额等价流（元/窗口），则 `U_* / market_value` 是 1/窗口 量纲，不是无量纲
3. 若 `U_*` 是市值占比流（无量纲），则 `capital_*` 也应是市值占比，但 `capital_* / P` 中 `P` 是价格（元/股），量纲不匹配

**后果：**
- `beta_norm` 的量纲依赖于 `U_*` 的定义，但文档声称其"无量纲或近似无量纲"
- 实际工程中可能因量纲混淆导致 `beta_norm` 数值不可比

**修复建议：**
1. 明确 `U_*` 的三种口径及其对应的 `beta_norm` 量纲：
   - `U_*_score`（无量纲）：`beta_norm` 量纲为 收益率/分数，不用于跨股比较
   - `U_*_amount`（元/窗口）：`beta_norm` 量纲为 收益率/(元/市值)，需统一市值单位
   - `U_*_mv_ratio`（无量纲）：`beta_norm` 为纯无量纲响应系数，可用于跨股比较
2. 仅在 `U_*_mv_ratio` 口径下输出正式的 `beta_norm` 和 `m_eff`
3. 增加 `beta_norm_unit` 字段，明确量纲
