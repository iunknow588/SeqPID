## 1. 文档目的

本文件定义比赛系统中的“资金运行模式分类模型”。

这里的 `pattern_type` 不应被理解为简单的价格形态标签，而应被理解为**资金运行模式识别结果**。  
模型目标不是扩充标签数，而是把日内成交、盘口、收盘位置和 PID 贡献统一解释成少量稳定模式。

当前官方提交目标更接近以下模式集合：

- `量化T0`
- `散户博弈`
- `尾盘突袭`
- `日内套利`
- `大单吸筹`

因此，模式分类模型需要完成两件事：

1. 识别资金运行模式
2. 给出可审计的证据与解释

## 2. 资金运行模式分类模型重构计划

### 阶段 1：统一目标口径

1. `pattern_type` 最终提交只输出五类资金运行模式：
   - `量化T0`
   - `散户博弈`
   - `尾盘突袭`
   - `日内套利`
   - `大单吸筹`
2. 当前内部标签不再直接作为提交标签。
3. 在 `label_dict.yaml` 中增加 `pattern_labels_submit`。

### 阶段 2：重构 Python 检测器

1. 将 `比赛系统/src/pattern_model.py` 从单体候选打分改成五个模式检测器。
2. 每个检测器输出 `score / evidence / confidence`。
3. 删除或替换当前 `del label_dict`，让模型读取提交标签配置。
4. 保留旧标签作为证据字段，不作为最终 `pattern_type`。

### 阶段 3：重构 Rust 检测器

1. 将 `src-rust/src/pattern_model.rs` 与 Python 规则对齐。
2. 抽取相同的五类模式评分逻辑。
3. 保证 Rust/Python 对同一样本输出一致。

### 阶段 4：引入 PID 二次修正

1. 使用 `dominant_type / dominant_intention` 辅助区分：
   - `量化T0`
   - `散户博弈`
   - `大单吸筹`
2. PID 只做修正，不直接替代原始模式检测。
3. 修正结果必须写入解释和诊断字段。

### 阶段 5：回归验证

1. 使用官方样例检查标签全集是否一致。
2. 使用 20260715 的 100 只样本重跑 Rust/Python。
3. 验证：
   - 输出标签全集只包含五类目标模式
   - Rust/Python `pattern_reco.csv` 一致
   - 典型样本解释能对应资金运行模式

## 3. 现状问题

当前实现存在三个问题：

1. `pattern_model` 主要是价格形态分类，不是资金运行模式分类。
2. Python 版本直接忽略 `label_dict`，Rust 版本也硬编码了内部标签池。
3. 现有标签更多是中间态证据，如 `盘中诱多 / 对倒拉升 / 压单吸货`，并非最终提交目标。

因此，现有模型的核心缺陷不是“少几个标签”，而是：

- 检测对象不对
- 证据聚合方式不对
- 最终输出口径不对

## 4. 目标定义

### 3.1 输出目标

最终输出仍保持原始字段：

```text
stock_code, transaction_date, pattern_type, pattern_explanation
```

但 `pattern_type` 必须是资金运行模式，而不是内部形态名称。

### 3.2 建议目标模式

| 模式 | 定义 |
| --- | --- |
| 量化T0 | 高频小单、快速回转、日内来回做差价 |
| 散户博弈 | 主导资金不强，小单混乱，噪声高、方向弱 |
| 尾盘突袭 | 尾盘集中放量，收盘段显著改变价格路径 |
| 日内套利 | 日内振幅大，但收盘不极端，反复高抛低吸 |
| 大单吸筹 | 大单持续承接，价格不一定暴涨，但筹码被吸收 |

---

## 5. 检测框架

模型采用“多检测器打分 + 规则裁决 + PID 二次修正”的三段式结构。

### 4.1 第一层：模式检测器

每个模式对应一个独立检测器，输出：

- `score`
- `evidence`
- `confidence`

检测器不是单一阈值，而是多个信号加权。

### 4.2 第二层：裁决器

若多个模式得分接近：

- 优先尾盘/收盘强特征
- 再看量化/散户主导性
- 再看日内振幅与回转特征

### 4.3 第三层：PID 修正

PID 不负责定义模式，只负责二次校验：

- `dominant_type`
- `dominant_intention`
- `hot_money_ratio / quant_ratio / retail_ratio`
- `tail_ratio / inertia / damping`

PID 只能修正模式倾向，不能直接替代模式检测。

## 6. 五类模式检测信号

### 5.1 量化T0

核心信号：

- `avg_trade_size` 偏小
- 买卖方向频繁切换
- `gross_flow / abs(net_flow)` 偏高
- 收盘不极端
- 日内振幅存在，但尾部不强

辅助证据：

- `quant_ratio` 高
- `burst_ratio` 中高
- `noise_ratio` 较高

### 5.2 散户博弈

核心信号：

- `retail_ratio` 高，或三类资金都不强
- 小单为主
- 盘口噪声大
- 方向性弱
- 收盘位置中性

辅助证据：

- `order_buy_ratio` 接近 0.5
- `explain_ratio` 低
- `noise_ratio` 高

### 5.3 尾盘突袭

核心信号：

- `tail_ratio` 高
- `last15_return` 显著
- 收盘前成交集中
- 收盘位置明显偏高或偏低

辅助证据：

- `close_strength` 极端
- `burst_ratio` 尾部放大

### 5.4 日内套利

核心信号：

- `intraday_range` 高
- `abs(close_return)` 不大
- 收盘靠近中性区
- 日内多次反转

辅助证据：

- `neutral_close_score` 高
- `mid_close_score` 高

### 5.5 大单吸筹

核心信号：

- 大单持续承接
- `close_strength` 偏高
- `bid_support` 强
- `ask_pressure` 不足以压垮价格

辅助证据：

- `hot_money_ratio` 高
- `large_order_score` 高
- `buy_bias_score` 高

## 7. 证据字段

建议每个样本保留以下证据字段，便于审计：

```text
pattern_primary_score
pattern_second_score
pattern_margin
pattern_evidence_top
pattern_evidence_secondary
pattern_source
pattern_pid_adjusted
```

推荐解释字段至少包含：

- 主要触发信号
- 次要触发信号
- PID 修正结果
- 是否发生规则覆盖

## 8. 现有内部标签的定位

当前内部标签建议只作为中间证据，不作为最终提交口径：

- `对倒拉升`
- `压单吸货`
- `盘中诱多`
- `连续小单推升`
- `分时脉冲`
- `集合竞价异动`
- `涨停板打开`

这些标签可以继续保留，但用途应改为：

1. 诊断
2. 解释
3. 中间层候选

## 9. 重构原则

### 8.1 不要只做标签映射

如果只做标签映射，模型仍然是旧的价格形态分类器。  
正确方向是把分类逻辑改成“模式检测器”。

### 8.2 Python / Rust 双端一致

两个实现必须共享：

- 同一套模式定义
- 同一套特征口径
- 同一套阈值配置
- 同一套裁决顺序

### 8.3 先检测，再压缩

内部可以保留更细标签，但最终提交必须统一到模式级标签。

---

## 10. 重构计划

### 阶段 1：标签口径统一

1. 明确最终 `pattern_type` 只输出五类模式
2. 保留内部细标签作为证据层
3. 抽取 `pattern_labels_submit`

### 阶段 2：检测器拆分

1. 将当前 `pattern_model` 改写为多检测器结构
2. 每个模式独立计算 score
3. 输出主次证据

### 阶段 3：PID 修正链路

1. 引入 `dominant_type / dominant_intention` 修正规则
2. 对 `量化T0 / 散户博弈 / 大单吸筹` 做二次修正
3. 仅在证据不足时启用降级规则

### 阶段 4：双引擎对齐

1. Python 和 Rust 保持规则等价
2. 用同一批样本做差异对齐
3. 增加单测和回归样本

### 阶段 5：验收

1. 输出标签全集符合官方提交口径
2. Rust/Python 结果一致
3. 典型样本解释合理

---

## 11. 建议验收指标

1. `pattern_type` 输出标签全集收敛到五类模式
2. 官方样例上的标签字段可直接对齐
3. 单日 100 样本 Rust/Python 输出一致
4. 典型样本的解释能对应到尾盘、套利、吸筹、博弈、T0 五类含义

---

## 12. 结论

`pattern_type` 的核心不是补标签，而是重建模式识别逻辑。  
当前模型应从“形态分类器”升级为“资金运行模式检测器”。
