## 三、重要风险项（🟡 编码阶段需重点关注）

### 3.1 【风险-01】`order_lifetime_ms` 字段的真实可得性存疑

**位置：** 详细设计 5.5节原始输入字段契约  
**问题描述：**

详细设计明确要求 `trades` 表包含 `order_lifetime_ms`（委托生存时长，毫秒），但：
- 大多数Level2数据供应商不提供逐笔成交对应的委托生存时间
- 该字段通常需要自行计算：委托时间 → 成交时间/撤单时间
- 若数据源只提供逐笔成交和逐笔委托两张表，需要撮合匹配才能计算寿命

**风险影响：** 高。若该字段无法直接从数据源获取，M2-01的订单分类逻辑（`order_lifetime_ms < 500` 或 `> 3000`）将无法执行。

**建议措施：**
1. **编码前必须确认数据源**：与数据供应商核实是否提供 `order_lifetime_ms` 或等效字段
2. **备选方案**：若不可得，需补充委托-成交匹配逻辑，计算近似寿命：
   ```python
   # 近似计算：逐笔委托表与逐笔成交表按 price+volume+time 匹配
   # 寿命 = 成交时间戳 - 委托时间戳（或撤单时间戳）
   # 注意：大单拆单、部分成交等复杂场景
   ```
3. **回退机制**：若寿命字段缺失，订单分类退化为仅按金额阈值（50万/10万）分类，文档需明确此降级路径

### 3.2 【风险-02】`rts_smooth` 中 `np.linalg.pinv` 替代矩阵求逆的精度损失

**位置：** 详细设计 4.3.5节  
**问题描述：**

```python
gain = p_filt @ np.linalg.pinv(p_pred_next + 1e-8 * np.eye(4))
```

使用伪逆 `pinv` 而非直接求逆，虽然增强了数值稳定性，但：
1. **精度损失**：`pinv` 基于SVD截断小奇异值，可能丢失有效信息
2. **条件数监控失效**：概要设计6.3节要求"若 `cond(P_{t+1|t})` 超过阈值需记录告警"，但 `pinv` 内部已做截断，外部无法感知条件数
3. **RTS理论最优性受损**：RTS平滑要求精确求逆，伪逆是近似

**风险影响：** 中。对大多数股票影响有限，但对高波动、强相关的参数（如φ_t和β_ch,t高度相关时）可能导致平滑轨迹偏差。

**建议措施：**
```python
# 方案：先尝试直接求逆，失败时回退到pinv并告警
def safe_matrix_inverse(p_pred_next, eps=1e-8):
    mat = p_pred_next + eps * np.eye(4)
    cond = np.linalg.cond(mat)
    
    if cond < 1e10:  # 条件数可接受
        return np.linalg.inv(mat), cond, "direct"
    else:
        # 条件数过高，使用pinv并记录
        return np.linalg.pinv(mat), cond, "pinv_fallback"

# 在FilterStep中记录求逆方式，用于后续诊断
```

### 3.3 【风险-03】`determine_agent_type` 中混合博弈型判定条件过于宽松

**位置：** 详细设计 4.4.5节  
**问题描述：**

```python
if attack_duration > thresholds["attack_40p"] and damping_duration > thresholds["damping_40p"]:
    ratio = max(attack_duration, damping_duration) / (min(attack_duration, damping_duration) + 1e-8)
    if ratio < 1.1:
        return AgentType.MIXED.value, 50.0, noise_ratio_median
```

问题：
1. **40%分位数阈值过低**：意味着超过40%的股票在某日会被判定为"攻击"或"阻尼"，这本身已不极端
2. **ratio < 1.1 过于严格**：要求攻击时长和阻尼时长几乎相等（差异<10%），实际市场中极少出现
3. **置信度硬编码50.0**：混合博弈型的置信度固定为50，低于单一主导型的动态计算，但未说明依据

**风险影响：** 中。可能导致混合博弈型样本过少（统计功效不足），或判定为混沌无效型过多，影响市场指数HMAI的稳定性。

**建议措施：**
1. 详细设计阶段补充模拟实验：用历史数据模拟判定，统计各类型占比
2. 调整ratio阈值至1.5或2.0，或改为动态基于历史分布
3. 混合博弈型置信度改为基于两者贡献度差异计算：
   ```python
   conf_mixed = 100 * (1 - abs(attack_duration - damping_duration) / (attack_duration + damping_duration))
   ```

### 3.4 【风险-04】全市场批处理内存峰值可能超标

**位置：** 详细设计 7.1节性能目标  
**问题描述：**

性能目标要求"批处理峰值内存 < 4GB"，但：
- 5000股 × 48窗口 × 多维度numpy数组，中间结果可能较大
- `multiprocessing.Pool` 默认会序列化参数传递，大对象序列化/反序列化开销高
- 每个子进程独立加载 `OfflineConfig`，若配置包含历史数据可能重复占用内存

**风险影响：** 中。在64并发下，若每个进程占用200MB，总内存可能超过12GB（远超4GB目标）。

**建议措施：**
1. **共享内存**：使用 `multiprocessing.shared_memory` 或 `joblib` 的 `memmapping` 模式共享只读数据
2. **Lazy加载**：`OfflineConfig` 按需加载，避免每个进程全量复制
3. **流式处理**：不等待全部5000股完成再统一写入，改为每完成一批（如100股）即写入并释放内存
4. **内存监控**：编码阶段增加 `psutil` 内存监控，超过阈值自动降并发
