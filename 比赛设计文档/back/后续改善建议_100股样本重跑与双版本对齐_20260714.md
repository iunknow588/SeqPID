# 后续改善建议：100股样本重跑与 Python/Rust 双版本对齐（2026-07-14）

## 1. 背景

基于最新 `C:\level-2-ana\data\百只股票样本.csv`，已重新执行 Python 与 Rust 两套比赛分析程序，并对正式提交结果、中间诊断文件、市场 PID 快照和回放报告进行了对比。

本次重跑暴露出一个重要差异：最新 100 股名单中有 3 只股票目录存在，但原始文件内容全为 `0x00` 空字节：

- `688687.SH`
- `688717.SH`
- `688819.SH`

Python 将这 3 只判定为无有效原始样本，并走市场中位水平补全路径；Rust 当前仍将其当作可建样本继续生成 48 个零值窗口。因此两版正式资金预测结果一致，但 PID 中间诊断和市场聚合统计存在轻微差异。

## 2. 当前结论

### 2.1 正式结果层面

`predict_result.csv` 两版 100 只股票完全一致。

`pattern_reco.csv` 仅 3 只股票的解释文本不同，模式类别一致，差异来自 Python 追加了补全说明：

> 缺失原始数据，按当日市场中位水平补全判断。

### 2.2 中间诊断层面

Python：

- 有效原始样本数：`97`
- 补全样本数：`3`
- `pid_daily_diag.csv`：`97` 行
- `pid_window_diag.csv`：`4656` 行

Rust：

- 有效原始样本数：`100`
- 补全样本数：`0`
- `pid_daily_diag.csv`：`100` 行
- `pid_window_diag.csv`：`4800` 行

差异原因不是名单读取错误，而是两版对“存在文件但文件内容无效”的处理策略不同。

## 3. 改善建议

### 3.1 统一空文件与全 NUL 文件判定规则

建议在设计文档和代码中明确：

1. 原始文件不存在，判定为 `missing_raw_file`。
2. 原始文件存在但大小为 0，判定为 `empty_raw_file`。
3. 原始文件存在但全部字节为 `0x00`，判定为 `null_filled_raw_file`。
4. 原始文件可读取但表头为空、乱码或无可识别字段，判定为 `invalid_raw_schema`。
5. 原始文件可读取但没有有效交易/行情记录，判定为 `no_effective_rows`。

以上情况均不应生成普通 PID 样本窗口，应进入缺失样本处理路径，并在诊断报告中显式记录。

### 3.2 Rust 对齐 Python 的缺失样本策略

建议修改 Rust 原始样本构建逻辑：

- 在读取每个原始 CSV 前增加二进制预检。
- 若文件全部为 `0x00`，直接返回无有效样本。
- 若三类原始文件均无有效数据，不生成 48 个零值窗口。
- 将该股票加入 `missing_symbols` 或 `incomplete_stock_dirs` 的诊断集合。
- 通过现有补全逻辑生成正式输出，保持提交文件 100 行完整。

目标是让 Rust 在以下指标上与 Python 对齐：

- `sample_count`
- `imputed_output_count`
- `missing_symbols`
- `pid_daily_diag.csv` 行数
- `pid_window_diag.csv` 行数
- `market_pid_snapshot.csv` 聚合口径

### 3.3 Python 保留并完善 CSV 读取防护

Python 已补充：

- `csv.field_size_limit(...)`，避免超长字段触发默认 128KB 限制。
- UTF-16 编码候选。
- NUL 表头检测。

建议继续完善为独立工具函数，避免 `data_loader.py` 与 `scheduler.py` 各自维护 CSV 读取逻辑。建议新增统一函数：

```python
read_csv_rows_with_quality(path, trade_date) -> tuple[list[dict], CsvQualityReport]
```

返回数据行的同时返回质量报告字段：

- `encoding_used`
- `file_size`
- `null_byte_ratio`
- `header_valid`
- `effective_row_count`
- `quality_status`
- `reason_code`

### 3.4 增加数据质量中间文档

建议新增一个批次级数据质量文件：

`raw_data_quality_report.csv`

推荐字段：

| 字段 | 含义 |
| --- | --- |
| `trade_date` | 交易日 |
| `symbol` | 股票代码 |
| `file_role` | `trade` / `order` / `quote` |
| `file_path` | 原始文件路径 |
| `file_exists` | 文件是否存在 |
| `file_size` | 文件大小 |
| `null_byte_ratio` | 空字节占比 |
| `encoding_used` | 实际读取编码 |
| `header_valid` | 表头是否有效 |
| `raw_row_count` | 原始行数 |
| `effective_row_count` | 有效行数 |
| `quality_status` | `ok` / `missing` / `empty` / `null_filled` / `invalid_schema` |
| `action` | `use_raw` / `impute` / `skip_diagnostics` |

该文件应作为 `100_stock_replay_report.md` 的数据来源之一。

### 3.5 补全样本也应进入日级诊断

当前 Python 对补全样本不生成 PID 诊断行，因此 `pid_daily_diag.csv` 只有 97 行。建议设计上明确两种选择：

方案 A：保持当前语义，仅真实有效样本进入 PID 诊断。

- 优点：PID 诊断只反映真实数据建模。
- 缺点：诊断行数与提交股票数不一致，需要报告解释。

方案 B：补全样本也生成一行日级诊断。

- `mode_name = imputed_market_median`
- `data_leakage_check = pass`
- `m_eff_rank_eligible = false`
- `submission_ready = true`
- `warnings = missing_raw_data_imputed`

建议采用方案 B。这样 `pid_daily_diag.csv` 可与 100 股提交清单一一对应，同时通过字段明确区分真实 PID 样本与补全样本。

### 3.6 回放报告应避免路径编码污染

当前部分报告中 `stock_list_file` 展示为乱码路径，例如 `鐧惧彧...`。建议：

- 所有报告写入统一使用 UTF-8。
- 路径字段写入前使用 `Path(...).resolve()` 后转字符串。
- 对 Markdown 报告补充编码回归测试，确保中文路径不被二次编码。

### 3.7 双版本一致性测试常态化

建议新增一个对比脚本：

`scripts/compare/python_rust_compare.ps1`

输入：

- Python 输出目录
- Rust 输出目录

输出：

- `python_rust_compare_report.md`
- `python_rust_diff_summary.csv`

比较范围：

1. `pattern_reco.csv`：股票集合、类别、说明文本。
2. `predict_result.csv`：股票集合、资金类型、资金意图。
3. `pid_daily_diag.csv`：股票集合、诊断字段。
4. `pid_window_diag.csv`：行数、窗口完整性、泄漏字段。
5. `market_pid_snapshot.csv`：市场宽度、PID 聚合指标。
6. `submit.zip`：内部文件存在性和官方字段。

比较结果应区分：

- `blocking_diff`：正式提交字段不一致。
- `diagnostic_diff`：中间诊断差异。
- `expected_diff`：编码、格式、解释文本等可接受差异。

## 4. 建议优先级

| 优先级 | 建议项 | 原因 |
| --- | --- | --- |
| P0 | Rust 识别全 NUL 原始文件并走缺失/补全路径 | 直接影响双版本诊断一致性 |
| P0 | 新增 `raw_data_quality_report.csv` | 让数据质量问题可审计、可复盘 |
| P1 | 补全样本进入 `pid_daily_diag.csv` | 让日级诊断与 100 股提交清单对齐 |
| P1 | 统一 Python CSV 读取工具函数 | 降低读取逻辑分叉和编码风险 |
| P1 | 修复 Markdown 报告中文路径编码 | 避免评审误读名单路径 |
| P2 | 建立 Python/Rust 自动对比脚本 | 支撑后续版本回归 |

## 5. 建议验收标准

完成上述 P0/P1 项后，针对最新 100 股名单重新执行，应满足：

1. Python 与 Rust 的 `predict_result.csv` 完全一致。
2. Python 与 Rust 的 `pattern_reco.csv` 至少在 `stock_code`、`transaction_date`、`pattern_type` 上一致。
3. 若原始文件全 NUL，两版均记录为 `null_filled_raw_file`。
4. `100_stock_replay_report.md` 中均显示：
   - `stock_universe_size = 100`
   - `missing_symbol_count = 3`
   - `imputed_output_count = 3`
5. `pid_daily_diag.csv` 若采用方案 B，应为 100 行，并能通过 `mode_name` 或 `warnings` 区分补全样本。
6. `market_pid_snapshot.csv` 的聚合口径两版一致。

## 6. 总结

本次差异并非模型预测逻辑冲突，而是数据质量边界处理不一致。正式预测结果已经高度一致，后续重点应放在“原始数据有效性判定”和“中间诊断口径统一”上。

建议优先让 Rust 对齐 Python 的缺失样本处理，并新增数据质量中间文档，使这类问题在批处理完成后可以被直接定位，而不是通过人工对比 PID 行数后再反推。
