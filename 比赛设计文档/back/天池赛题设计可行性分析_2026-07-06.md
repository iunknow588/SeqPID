# 天池赛题一：比赛设计文档可行性分析报告

**分析对象版本：**
- `比赛概要设计说明书.md` V1.1
- `比赛详细设计说明书.md` V1.1

## 结论

**该比赛设计文档整体可行，且在 V1.1 中已吸收前一轮关键风险建议，但仍存在3个高优先级闭环事项。** 文档架构清晰，模块解耦合理，双引擎设计（统计特征+可解释状态）有竞争力。当前主要待验证点集中在：数据可得性、Task 1 聚类稳定性、基线版到优化版的阶段切换管理。

## 二、架构设计评估

### 2.1 总体架构（七层设计）

```
数据接入层 → 窗口切分层 → 行为统计特征层 → 可解释状态层 → 任务建模层 → 提交导出层 → 调度复现层
```

**评估：分层合理，职责清晰。**

- 数据接入层（M0）：支持原始重算和参考特征直读两种模式，灵活
- 窗口切分层（M1）：48个5分钟窗口 + 派生窗口，与现有状态空间设计兼容
- 双引擎设计：统计特征引擎（85维）+ 可解释状态引擎（KF+RTS），有差异化竞争力

### 2.2 模块解耦度

| 模块 | 职责 | 耦合度 |
|------|------|--------|
| Task 1（pattern_model.py） | 交易模式聚类与归类 | 低 - 仅依赖特征层 |
| Task 2（capital_model.py） | 参与者类型与意图识别 | 低 - 仅依赖特征层 |
| 特征层（reference/state） | 特征构建 | 低 - 仅依赖数据层 |
| 导出层（exporter.py） | CSV生成与打包 | 极低 - 仅依赖结果层 |

**评估：Task 1 与 Task 2 共享底层特征但建模解耦，符合竞赛开发最佳实践。**

## 三、关键风险点

### 风险1：数据可得性未验证 ⚠️ 高

| 数据项 | 设计文档假设 | 当前设计响应 | 验证状态 |
|--------|------------|-------------|---------|
| 逐笔成交 | 直接读取 | 已纳入 schema 探针 | ❌ 未验证文件格式 |
| 逐笔委托 | 直接读取 | 已纳入 schema 探针 | ❌ 未验证文件格式 |
| 逐笔撤单 | 直接读取 | 已纳入 schema 探针 | ❌ 未验证文件格式 |
| 十档盘口快照 | 直接读取 | 已纳入 schema 探针 | ❌ 未验证字段契约 |
| order_lifetime_ms | 核心字段 | 已设计回退路径 | ❌ 未验证可得性 |
| 85维参考特征 | 可选直读 | 已设计直读/重算双模式 | ❌ 未确认是否提供 |

**影响：** 若 order_lifetime_ms 缺失，Task 2 的游资/量化初筛将退化为金额阈值版本，分类精度可能显著下降。

**V1.1 已补强：**
- 新增 `schema_probe.py`
- 新增 `probe_input_schema()`
- 要求第一周产出 `schema_probe_report.md`

**建议：** 立即向赛题方或实际数据源确认数据 schema，特别是：
1. 四类原始数据的文件格式（CSV/Parquet/Binary？）
2. 字段命名规范是否与文档中 RawTrade/RawOrder 等定义一致
3. order_lifetime_ms 是否可得
4. 85维参考特征是赛前提供还是需要自行计算

### 风险2：聚类稳定性存疑 ⚠️ 中高

**问题：** Task 1 采用“离线聚类原型 + 在线最近原型归类”方案，但：

1. **原型库固化风险**：HDBSCAN/KMedoids 聚类结果可能随训练样本变化，原型库版本管理复杂
2. **距离函数设计复杂**：D = w1·DTW + w2·Wasserstein + w3·Summary，权重 w1/w2/w3 需大量调参
3. **模式标签主观性**："大单吸筹"、"尾盘突袭"等标签的ground truth难以获取，聚类质量无法客观评估

**影响：** Task 1 的评估指标是"类内聚合度与类间区分度"，若聚类不稳定，可能导致：
- 不同日期同类股票被分到不同模式
- 模式解释（pattern_explanation）质量参差不齐

**V1.1 已补强：**
- 新增 `pattern_margin_threshold`
- 新增 `fallback_pattern_rule()`
- 明确 HDBSCAN 不稳定时可回退到 `KMeans/KMedoids`

**建议：** 
- 优先用历史数据验证聚类稳定性（连续N日同股票的模式一致性）
- 考虑引入时序约束（如昨日模式作为今日先验）提升稳定性
- 准备规则兜底方案（当聚类置信度低时，回退到基于阈值的模式判断）
- 将 `pattern_stability_report.md` 作为第三阶段冻结前强制交付物

### 风险3：开发周期与里程碑匹配度 ⚠️ 中

文档建议四阶段开发，并在 V1.1 中新增“里程碑验收门”：

| 阶段 | 内容 | 预估工作量 | 风险 |
|------|------|-----------|------|
| 第一阶段 | 数据接入 + 85维特征 + CSV导出 | 3-5天 | 数据格式确认后风险低 |
| 第二阶段 | 状态引擎 + Task 2 基线模型 | 5-7天 | 标签数据获取难度不确定 |
| 第三阶段 | Task 1 聚类 + submit.zip | 5-7天 | 聚类调参可能耗时超预期 |
| 第四阶段 | 压测 + 异常演练 + 复现材料 | 3-5天 | 依赖前三阶段稳定性 |

**总预估：16-24天**

**问题：** 
- 若比赛时间紧迫（如仅剩2-3周），第三阶段聚类调参可能压缩第四阶段测试时间
- Task 2 的"规则种子 + 弱监督扩展"方案依赖高置信度种子样本，若种子质量不高，整个分类器基础不牢

**V1.1 已补强：**
- 性能目标拆分为“基线版 / 优化版”
- 新增 `baseline_mode`
- 新增“第一阶段结束即可提交”的最低交付门槛

**建议：** 
- 采用敏捷迭代：先出可运行基线（第一阶段结束即可提交），再逐步优化
- Task 2 优先用规则引擎出初版结果，确保有输出，再逐步替换为模型
- Task 1 先用简单聚类（如K-Means）出初版，再升级为HDBSCAN+KMedoids
- 明确每阶段的“冻结条件”，避免第三阶段拖慢整体提交流程

## 四、技术方案亮点

### 4.1 双引擎设计

统计特征引擎（85维参考特征）+ 可解释状态引擎（KF+RTS）的组合有差异化优势：
- 85维特征覆盖赛题参考体系，确保不遗漏关键行为信号
- 状态解释特征（惯性、阻尼、游资冲击、量化供给）提供可解释性，便于调试和复现审核

### 4.2 降级设计完善

文档第11章异常与降级设计覆盖了：
- 数据缺失（撤单/盘口/order_lifetime_ms）
- 模型异常（原型库/分类器加载失败）
- 提交异常（字段顺序/行数不一致）
- 聚类异常（HDBSCAN 不可用时回退基线聚类）

**评估：降级策略成熟，符合生产级系统要求。**

### 4.3 可复现性保障

- 配置快照 + 特征版本 + 模型版本 + 原型库版本
- 数据缺失与降级日志
- 输出文件 MD5

**评估：满足赛题复现审核要求。**

## 五、性能目标评估

V1.1 已将性能目标拆分为两阶段：

| 指标 | 基线版目标 | 优化版目标 | 评估 |
|------|-----------|-----------|------|
| 单股单日全流程 | < 1.0秒 | < 0.5秒 | 更合理，首版可先按 1.0 秒验收 |
| 5000股全市场日终 | < 35分钟 | < 20分钟 | 更贴近真实开发节奏 |
| 提交文件生成 | < 30秒 | < 15秒 | 合理 |
| 重跑单日 | < 15分钟 | < 10分钟 | 合理 |

**评估：** 性能目标在 V1.1 后已明显更合理，但优化版目标仍需依赖向量化与并行能力验证。

**建议：**
- 特征计算向量化（NumPy/Pandas）
- 多进程并行（文档已建议process_count: 32）
- 先按基线版目标验收，再逐步推进优化版目标

| 维度 | 评分 | 说明 |
|------|------|------|
| 赛题对齐度 | ✅ 优秀 | 完全覆盖赛题要求，无遗漏 |
| 架构合理性 | ✅ 优秀 | 七层分层清晰，模块解耦 |
| 技术可行性 | ✅ 良好 | 双引擎设计合理，且已增加回退与稳态约束 |
| 数据可得性 | ⚠️ 待验证 | schema 探针已设计，但真实数据尚未验证 |
| 开发周期 | ⚠️ 可控 | 基线优先后风险下降，但仍需严格里程碑管理 |
| 性能目标 | ⚠️ 可控 | 已拆为基线版/优化版，需实测验证 |
| 降级健壮性 | ✅ 优秀 | 异常覆盖全面，新增聚类回退 |
| 可复现性 | ✅ 优秀 | 版本管理+审计日志完善 |

### 总体结论

**该设计文档可行，且 V1.1 已把主要建议吸收进设计正文，但仍需在编码前闭环3个关键事项：**

1. **立即确认数据schema**：特别是order_lifetime_ms可得性、85维参考特征是否提供、原始数据文件格式
2. **优先构建基线**：第一阶段结束即产出可运行版本，确保有提交能力，再逐步优化聚类和分类模型
3. **预留聚类调参时间**：Task 1 的聚类稳定性是最大不确定性，建议预留充足时间验证和迭代

若以上3点闭环，该方案有竞争力，双引擎设计在可解释性和差异化上有优势。
